# Vendored cross-implementation test data

Everything `tests/test_cross_implementation.py` reads. Together these make that
suite self-contained: it needs no PINT *source checkout* beside the repo, and no
tempo2/libstempo installation.

| directory | contents | size |
|---|---|---|
| `pint_inputs/` | 37 `.par` / `.tim` files, verbatim from `PINT/tests/datafile` | 10.5 MB |
| `pint_goldens/` | upstream reference residuals (`.tempo2_test`, `.tempo_test`, `.libstempo`) | 1.6 MB |
| `tempo2_goldens/` | our own tempo2 references, with provenance headers | 1.8 MB |

## Why vendored rather than fetched

`pint.config.examplefile()` reads `pint/data/examples` in the installed
`pint-pulsar` wheel, which holds **23 files — only 5 of the 30 inputs used
here**. The remainder live solely in `PINT/tests/datafile`, which is not
packaged; the goldens are not packaged either. So the wheel cannot supply them
and the only alternative was depending on a sibling checkout, which made the
whole suite skip silently for anyone who had merely pip-installed PINT.

These are frozen upstream fixtures — write-once, never edited — so they cost
history size only at the initial commit.

## Provenance

`pint_inputs/` and `pint_goldens/` are unmodified copies from the PINT
repository at commit `13245864` (`git describe`: 1.1.4-85-g13245864), BSD
3-Clause. PINT's licence is reproduced here as `LICENSE.PINT.md`, as that
licence requires of redistributions.

`tempo2_goldens/` is generated by `tools/gen_tempo2_goldens.py` against
tempo2 2021.07.1; each file records the tempo2 and libstempo versions plus a
SHA-256 over `$TEMPO2/clock`. Unlike the upstream goldens — which record bare
numbers — that header is what makes a disagreement diagnosable.
